# 第 1 课：书签是什么，以及如何生成与保存

## 1. 为什么需要 Bookmark Data？

用户在 **「文件」App** 或通过 **UIDocumentPicker / SwiftUI `fileImporter`** 选了一个**文件夹或文件**后，你拿到的是一个 **`URL`**。在 iOS 沙盒里，这种访问往往是 **临时的、受系统控制的**（安全作用域）。

若只想在**当前这次前台会话**里读一下，可能够用了；但若希望：

- **下次启动 App** 仍能打开**同一位置**，或  
- 在进程重启后**不必每次都让用户再选一遍**，

光靠把路径 **字符串** 存进 `UserDefaults` **通常不够**：路径可能变化，更关键的是 **授权不会跟着一串 path 走**。

**Bookmark Data** 是 `URL` 提供的一种 **不透明二进制数据**：系统把「如何再次定位并（在允许时）访问该资源」编码进去。你把这段 `Data` 存到 **Keychain / 文件 / 数据库** 任意地方；之后用 `URL(resolvingBookmarkData:…)` **解析**回 `URL`，再按规则做 **security-scoped** 访问。

> **牢记**：书签解决的是 **「在授权仍然有效的前提下，持久化引用」**；不能替代「用户永远不卸载、永远不撤销授权」这类产品假设（第 2 课会展开）。

## 2. 核心 API（概念）

### 2.1 创建书签

```swift
let data = try url.bookmarkData(
    options: …,  // 平台相关，见下文
    includingResourceValuesForKeys: nil,
    relativeTo: nil
)
```

- 返回值 **`Data`** 即要持久化的 **bookmark data**。  
- **`options`** 在 **macOS** 与 **iOS** 上**可用项不同**（这是面试与真机 bug 的高发区）。

### 2.2 解析书签

```swift
var isStale = false
let url = try URL(
    resolvingBookmarkData: data,
    options: …,
    relativeTo: nil,
    bookmarkDataIsStale: &isStale
)
```

- 成功则得到一个 **`URL`**，通常还需要 **`startAccessingSecurityScopedResource()`**（对需要安全作用域的场景）。  
- **`isStale`**：表示「这段 bookmark 数据已过时，但系统仍尽力给了你一个 URL」——**第 2 课**专门讲如何维护。

## 3. 平台差异（务必分开记）

### 3.1 macOS

- 对**用户选定目录**做可跨重启访问的书签时，创建/解析常搭配 **`.withSecurityScope`**（在 Swift API 里对应 `NSURL.BookmarkCreationOptions` / `BookmarkResolutionOptions`）。  
- 访问文件前：**`url.startAccessingSecurityScopedResource()`**，用完后 **`stopAccessingSecurityScopedResource()`**（注意配对，避免泄漏）。

### 3.2 iOS

- 文档与真机行为上：**`BookmarkCreationOptions.withSecurityScope` / 解析侧的同名项在 iOS 上不可用**（你只能在 Swift 里对某些 option 使用 **空数组** 或文档允许的选项，与 macOS 示例**不要照抄**）。  
- 从 Document Picker 拿到的 URL，仍然需要在访问资源前 **`startAccessingSecurityScopedResource()`**（与「options 里有没有 withSecurityScope」不是同一件事）。

**本课结论**：写跨平台封装时，用 **`#if os(macOS)` / `os(iOS)`** 分开 options，**不要**把 macOS 書签代码原样粘到 iOS。

## 4. 典型流程（iOS：选文件夹 → 存书签）

下面是与「第 2 课」衔接的最小逻辑示意（**非完整 App**）。

**（1）用户从选取器得到 `url`：**

```swift
guard url.startAccessingSecurityScopedResource() else {
    // 无法建立访问，提示用户
    return
}
defer { url.stopAccessingSecurityScopedResource() }  // 若此处只生成书签就结束，按你的生命周期配对
```

实际项目里，你可能在持有该目录期间**长期**保持 `startAccessing`，直到用户更换目录再等；**配对**要一致。

**（2）生成 bookmark `Data` 并持久化：**

```swift
do {
    let bookmark = try url.bookmarkData(
        options: [],  // iOS：按当前 SDK 使用允许的选项；不要填 macOS 专用 .withSecurityScope
        includingResourceValuesForKeys: nil,
        relativeTo: nil
    )
    // 写入 Keychain / 文件 等
} catch {
    // 处理错误
}
```

**（3）展示目录内容**仍在**已开启安全作用域**的同一个 `url` 上进行即可。

## 5. 持久化存在哪里？

常见选择：

| 存储 | 说明 |
|------|------|
| **Keychain** | 适合存**体积不大**的 blob；注意卸载应用后钥匙串项通常会被清掉，且**不等于**「授权跨重装延续」。 |
| **UserDefaults / 文件** | 也可存 `Data`；安全性与备份策略按产品要求选型。 |

**本课不要求**你实现 Keychain 细节，只需明确：**bookmark  bytes 存哪儿**与 **系统是否仍允许解析**是两件事。
